# Data Quality Findings & Engineering Decisions

This document captures data issues, test failures, and the decisions made to resolve them.
It is intended to support a transparent discussion of the project during a technical interview.

---

## 1. Source Dataset Limitations

### Dataset: Data Science Job Salaries (ai-jobs.net, Kaggle)

| Metric | Value |
|--------|-------|
| Total raw rows | ~3,900 |
| After EU country filter | ~300 |
| After role classification filter | ~116 |

**Why so few rows?**
- The dataset is a global salary survey skewed toward North America.
- European respondents are underrepresented (~8% of total).
- The role classification macro (`classify_role_family`) filters out roles that
  don't match data/ML/AI patterns ‚Äî catching only ~40% of EU rows.

**Interview talking point:**
> "In a production context I would augment this with the Stack Overflow Developer Survey
> (~65k respondents, stronger EU coverage) or combine multiple years of ai-jobs.net data.
> The architecture is deliberately dataset-agnostic ‚Äî swapping the source only requires
> updating the staging model and source YAML."

## 2. Test Failures Encountered

### 2.1 Surrogate Key Uniqueness ‚Äî 4 failures ‚Üí fixed

**Tests failed:**
- `unique_stg_survey_responses_response_id` ‚Äî 29 duplicates
- `unique_int_roles_classified_response_id` ‚Äî 2 duplicates
- `unique_int_salary_filtered_response_id` ‚Äî 2 duplicates
- `unique_fact_compensation_response_id` ‚Äî 2 duplicates

**Root cause:**
The surrogate key was generated from business columns only
(`work_year`, `experience_level`, `job_title`, `salary`, etc.).
The source dataset contains rows where **all business columns are identical**
(e.g., two Senior Data Engineers in Germany earning the same salary in the same year),
causing hash collisions.

**Fix:**
Added `row_index` (the CSV's unnamed first column ‚Äî a sequential row number)
to the `generate_surrogate_key()` call in `stg_survey_responses.sql`:

```sql
{{ dbt_utils.generate_surrogate_key([
    'row_index', 'work_year', 'experience_level', ...
]) }} as response_id
```

**Why `row_index` is valid here:**
The source is a static CSV loaded once. The row index is stable across loads
of the same file, making the key deterministic and reproducible.

**Residual risk:**
If the source CSV is re-sorted or regenerated by the data provider,
row indices would shift and all surrogate keys would change ‚Äî triggering
a full merge rewrite on the incremental model. Acceptable for a static dataset;
for a live feed, a natural business key would be preferable.

### 2.2 Stale Incremental Data ‚Äî 1 failure ‚Üí fixed with `--full-refresh`

**Test failed:**
- `unique_fact_compensation_response_id` ‚Äî 2 duplicates (persisted after fix 2.1)

**Root cause:**
`fact_compensation` is an incremental model using `merge` on `response_id`.
After changing the surrogate key logic, the old rows (built with the pre-fix key)
remained in the table. The merge added new rows with new keys alongside the old ones,
creating duplicates for 2 records.

**Fix:**
```bash
dbt run --select fact_compensation --full-refresh
```

**Learning:**
Any change to the surrogate key logic of an incremental model requires a full refresh.
This is a known operational risk with incremental models ‚Äî documented in the README
under "Key design decisions."

## 4. mart_salary_by_region ‚Äî Low Row Count

The mart applies a `WHERE respondent_count >= 5` filter to suppress unreliable aggregates.

**Result:** Only **6 rows** pass this threshold out of all possible country √ó role √ó seniority combinations.

**Why this is correct behaviour, not a bug:**
Showing a "median salary" for a single respondent would be misleading.
The filter is a deliberate data quality gate ‚Äî the same approach used by
compensation benchmarking platforms (e.g., Levels.fyi suppresses cohorts < 5).

**What this means for the dashboard:**
The Looker Studio dashboard will show limited combinations.
This is an honest reflection of dataset coverage, not an engineering failure.

**How to improve:**
- Load multi-year data (2020‚Äì2024 in one table) to increase row count
- Add the Stack Overflow survey as a second source and union at the staging layer

## 6. What Would Be Different in Production

| Area | Current (demo) | Production |
|------|---------------|------------|
| Dataset size | ~3,900 rows | SO Survey (~65k) + multi-year ai-jobs.net |
| Surrogate key | `row_index` + business columns | Natural business key from source system |
| Incremental watermark | `_bq_loaded_at` (load timestamp) | `work_year` or a feed `posted_date` |
| Outlier removal | P1/P99 per country | P1/P99 per country + role (more granular) |
| respondent_count threshold | 5 | Tunable per dashboard use case |
| CI secrets | Not configured | `GCP_SA_KEY` + `DBT_BQ_PROJECT` in GitHub secrets |
